# Design: 信息科技学生学习平台重构方案

生成时间：2026-03-30  
状态：DRAFT  
模式：Builder（基于当前上下文推定为“真实学校交付项目”，不是纯练手 Demo）  
范围：对现有 LearnSite/学习网站进行完整功能梳理，并给出 Vue3 + FastAPI 重构方案  

## 1. 项目目标

本项目的目标不是做一个“类似学习网站的全新产品”，而是：

1. 以现有站点 `http://10.230.39.247/` 为业务蓝本，完整继承其核心教学流程、角色体系和管理逻辑。
2. 将原 ASP.NET WebForms / aspx 后端重构为 `FastAPI`。
3. 将前端重构为 `Vue 3 + TypeScript + Element Plus`，并显著提升视觉表现与可用性。
4. 在保留原站教学闭环的前提下，补上现有系统缺失的正式课程体系、教师管理员权限合并、主题系统和 AI 智能体能力。
5. 第一阶段允许使用 `SQLite`，但数据结构必须从第一天就兼容未来迁移到 `PostgreSQL`。

## 1.1 已确认范围

以下内容已经由项目发起方确认：

1. 第一阶段只覆盖线上站点当前明确在用的功能。
2. 开源仓库中“已存在但线上未明确启用”的增强模块，只作为分析参考，不纳入首版范围。
3. 学生正式登录入口以 `http://10.230.39.247/index.aspx` 为准。
4. 首版不迁移旧系统历史数据，只做新平台。

## 3. 我对项目的理解

你要做的不是普通“课程网站”，而是一个面向初中信息科技课堂的教学平台。它至少要覆盖三类真实用户：

1. 学生：登录学习、完成任务、上传作品、参加测验、查看资源、参与互动。
2. 教师：备课、上课、控制课堂、查看作品、签到、管理学生、组织测验、管理资源。
3. 管理员：配置学校基础信息、班级、教师、导入学生、升班、备份、维护系统设置。

同时你已经明确提出了 8 个升级方向：

1. 页面要更适合初中生，视觉上丰富、自然，并支持主题切换。
2. 后端从 aspx 改成 FastAPI。
3. 前端改为 Vue3，并使用 Element Plus 等流行框架。
4. 管理员与教师合并为一个后台，只是教师是否拥有管理员权限不同。
5. 数据库先用 SQLite，成熟后迁移 PostgreSQL。
6. 增加两类智能体：
   - 课时智能体：由教师创建，绑定到某个课时/学案。
   - 通用智能体：全站悬浮入口，连接 OpenAI 兼容接口。
7. 备课添加学案时，必须能选择教材中的“哪一课”，后台要维护完整课程体系。
8. 文档必须非常详细，方便给其他团队继续开发与交付。

## 4. 现有系统功能地图

## 4.1 入口与角色

现有系统采用三种登录/入口形态：

1. 学生入口：
   - 正式登录地址 `index.aspx`
   - 根路径 `/` 当前也会落到学生首页登录界面，但产品定义以 `index.aspx` 为准
   - 首页提供学生登录、学员注册、学号查询、课堂守则、教师平台入口
2. 教师入口：
   - `teacher/index.aspx`
3. 管理员入口：
   - 与教师使用同一个登录页 `teacher/index.aspx`
   - 登录后根据账号跳转到 `manager/`

这说明当前系统虽然有“三种角色”，但登录入口本质上已经部分复用，未来合并教师/管理员后台是合理的。

## 4.2 学生端公开可见功能

本次实测确认了以下公开页面：

1. 登录首页
   - 学号登录
   - 密码登录
   - 学员注册
   - 学号查询
   - 课堂守则
2. 学员注册页 `student/register.aspx`
3. 学号查询页 `student/mynum.aspx`
4. 课堂守则页 `student/myrule.aspx`

其中 `student/mynum.aspx` 当前会直接展示学生姓名与学号对应关系，这是一个明显的隐私/安全风险，后续不建议原样继承。

## 4.3 学生端登录后功能

本轮已使用你提供的学生账号实际进入学生端，并对线上页面逐页遍历。以下内容应视为“已验证的线上功能”，优先级高于源码推断。

1. 学习中心首页 `student/myinfo.aspx`
   - 顶部主导航为：我的学案、我的作品、打字宝典、常识积累、在线资源、我的网盘。
   - 页面按“未学学案 / 已学学案”展示课程，并可直接进入 `showcourse.aspx?cid=*`。
   - 首页同时显示当日签到同学、学生基本信息、组队入口。
2. 学案详情页 `student/showcourse.aspx`
   - 展示课时标题、任务列表、返回入口。
   - 某些课程页同时展示学习目标、课件下载、外部资料链接。
   - 已实测课程既包含当前 AI 单元，也包含上学期历史课时，说明该页承担跨学期课程访问。
3. 阅读/说明类页面 `student/description.aspx`
   - 用于展示长文说明、活动指南、阅读材料。
   - 支持外链、课件/文档附件下载。
   - 支持“已阅读”确认按钮。
4. 通用任务执行页 `student/showmission.aspx`
   - 当前线上大量活动通过该页承载作品上传。
   - 页面包含任务标题、前后任务导航、作品互评入口、我的网盘入口、文件上传控件。
   - 已提交作品可再次下载并重新提交。
   - 某些任务存在前置约束，例如“请先提交前面作品”。
5. 代码/创作任务页 `student/program.aspx`
   - 历史在线课程已实测存在 Python 编程任务。
   - 页面展示任务说明、继续创作入口、作品互评入口。
   - 说明学生端不仅是文件上传，也需要承载浏览器内创作型任务。
6. 作品中心 `student/mywork.aspx`
   - 按学案/活动列出历史提交记录。
   - 每条记录包含作品下载、学分/加分/鲜花、年级、学期等列。
   - 支持分页。
7. 作品互评 `student/myevaluate.aspx`
   - 以作品墙方式展示全班作品。
   - 显示作品总数、可投次数、我的得票、互评得分。
   - 学生可为作品投票/推荐。
8. 常识测验 `student/myquiz.aspx` 与 `student/quizrank.aspx`
   - 首页显示平均成绩并提供“开始测验”入口。
   - 排行页显示当日班级排行榜和未测验名单。
9. 打字训练 `student/myfinger.aspx` 与 `student/allfinger.aspx`
   - 当前在线主入口为英文打字训练。
   - 支持不同词库级别、历史成绩展示、排行榜。
   - 页面还链接推箱子游戏、太空打字游戏等训练/激励玩法。
10. 网盘 `student/groupshare.aspx`
   - 支持“我的网盘 / 小组网盘”切换。
   - 页面展示已用容量、文件数、状态信息。
   - 顶部导航与任务页中都可直接打开网盘弹层。
11. 个人资料中心 `profile/*`
   - 已实测入口包括：基本信息、小组、签到记录、密码修改、班级修改、相片上传、性别修改、姓名修改。
   - 说明学生端包含自助资料维护能力，而不只是课堂学习页。
12. 在线资源 `student/myfile.aspx`
   - 当前线上入口仍在导航中，但实测页面内容基本为空白。
   - 这说明资源模块属于保留功能位，首版应保留模块框架，但可以按“当前内容空缺”处理数据展示策略。
13. 当前线上已确认的任务形态
   - 课程/学案浏览
   - 阅读材料与说明页
   - 附件下载与外部链接跳转
   - 截图/图片作品上传
   - 浏览器内代码任务
   - 作品互评
   - 常识测验
   - 打字训练
   - 网盘与小组协作
   - 个人资料维护

补充观察：

1. 学生首页姓名链接当前直接跳到 `teacher/studentwork.aspx?Snum=...`，这是典型旧系统路径复用，新平台不建议保留这种跨角色 URL 结构。
2. `showcourse.aspx?cid=22` 的“心理测评”页面直接展示多名学生的测评账号与密码，这是比公开学号查询更严重的隐私问题，重构时必须移除。
3. 源码里还能看到调查、讨论、聊天、游戏、更多创作工具等扩展能力，但本次未在当前线上主链路中实测到明确入口，按照已确认范围不纳入首版。
4. 结论：学生端不是“单页课程浏览器”，而是一个包含学习、提交、互评、测验、训练、网盘、个人中心的完整学习工作台。

## 5. 现有系统的关键问题

## 5.1 产品与体验问题

1. 视觉风格对初中生吸引力不足，整体仍偏旧式 WebForms 风格。
2. 学生端、教师端、后台端视觉语言不统一。
3. “上课”页功能很多，但交互密度高、理解门槛高，缺少现代化的信息层级。
4. 备课页的“学案”是核心，但课程体系并不是正式实体，查找和复用逻辑弱。
5. 模块很多，但导航结构没有现代系统级信息架构。

## 5.2 架构问题

1. WebForms 与页面后置逻辑耦合重，难于维护。
2. 业务通过页面跳转、隐藏字段、Cookie、表单回发驱动，测试和重构成本高。
3. 许多功能通过“页面复制 + 参数变体”扩展，缺少统一领域模型。
4. 教师和管理员后台人为分裂，但底层实际已共享很多概念。

## 5.3 数据与配置问题

1. 课程体系不完整，只能通过“分类 + 年级 + 课节”粗粒度组织学案。
2. 学案与教材课时不是强绑定关系，无法稳定支持“八年级-第一单元-第 2 课”这种结构。
3. 很多课堂开关直接挂在 `Room` 表布尔字段上，后续扩展容易失控。
4. 文件资源、作品、任务、题库、课程、班级之间关系复杂，但没有明确模块边界。

## 5.4 安全与隐私问题

1. `website.xml` 中存在系统级配置集中存储的设计。
2. 源码快照中可见 SMTP 凭据和 AI 平台密钥被明文存放在配置文件中，这是高风险做法。
3. 学号查询页存在公开枚举学生姓名与学号的风险。
4. 旧系统大量依赖 Cookie 与页面跳转，鉴权边界不清晰。

## 5.5 交付与迁移风险

1. 功能总量很大，不能用“几个页面 + 几张表”理解。
2. 如果直接按旧页面逐页照搬，新的 Vue3 + FastAPI 版会把旧系统的问题一起复制过来。
3. 如果完全重写而不做分阶段计划，范围会过大。
4. 学生登录链路已经表现出历史复杂性，说明旧系统中存在未显性的状态耦合。

## 6. 设计前提（Premises）

以下前提是我在本轮分析后形成的判断，建议下一轮由你确认：

1. 新系统要“业务全量继承”，但不需要 HTML 页面逐像素复刻。
2. 教师与管理员在新系统中应该是同一个后台，只是权限集合不同。
3. “课程体系”必须成为一等公民，不能继续只靠分类、年级、课节来模拟。
4. 学案是“教学执行方案”，课程体系是“教材/教研结构”，这两个概念必须分开。
5. AI 能力必须通过统一 OpenAI-compatible Provider 抽象接入，不要写死某一家模型平台。
6. 第一阶段虽然使用 SQLite，但所有模型、迁移和查询方式都要按 PostgreSQL 兼容方式来设计。
7. 学生端需要主题系统，而且主题是产品能力，不是后期装饰。
8. 历史系统的弱安全设计不能照搬，尤其是明文密钥、公开学号枚举、脆弱登录模式。

## 7. 方案备选（Mandatory Alternatives）

## Approach A：逐页功能复刻

Summary：按现有页面结构一页一页重做，菜单、入口和流程尽量保持一致。  
Effort：L  
Risk：高

Pros：

1. 最容易对照现有站点验收。
2. 需求沟通成本最低。
3. 适合短期“替换旧站”思维。

Cons：

1. 会把旧系统的信息架构与历史问题整体复制到新技术栈。
2. 难以优雅地接入课程体系、主题系统、统一权限、AI 智能体。
3. 长期会得到一个“Vue 外壳下的旧站”。

Reuses：

1. 菜单命名
2. 页面流程
3. 原有功能边界

## Approach B：领域重构式重建

Summary：先抽象领域模型，再围绕课程、学案、课堂、作品、测验、资源、用户、权限等模块重建系统。  
Effort：XL  
Risk：中

Pros：

1. 能从根上解决课程体系、教师管理员合并、AI、主题和权限模型问题。
2. 后续扩展性最好。
3. 更适合给团队长期维护。

Cons：

1. 前期设计成本高。
2. 如果没有阶段拆分，首版交付周期会拉长。
3. 与旧站逐页对照时需要更强的验收文档。

Reuses：

1. 旧站真实业务闭环
2. 旧站数据概念
3. 教学流程与角色边界

## Approach C：分阶段兼容式重构

Summary：以领域重构为目标，但按“可替换模块”分阶段交付，先做统一身份、课堂中控、课程体系、学案与任务，再逐步替换作品、资源、测验、打字、后台设置等模块。  
Effort：L~XL（按阶段展开）  
Risk：低到中

Pros：

1. 最适合本项目的真实规模。
2. 既不会被旧页面绑死，也不会一次性把范围做成失控大项目。
3. 可以把“核心替代模块”和“仓库里已有扩展模块”拆成不同阶段。

Cons：

1. 需要更细的阶段定义与验收标准。
2. 数据迁移与旧系统对照工作会更细碎。
3. 文档必须非常详细，否则团队容易在阶段切换时失焦。

Reuses：

1. 原站功能地图
2. 原站业务词汇
3. 原站数据
4. 原站模块优先级

**RECOMMENDATION：Choose Approach C，因为它最适合这个项目的真实体量。**  
它以领域重构为目标，但用分阶段交付控制风险，既能完成“完整复刻”，也能真正把课程体系、主题系统、AI 智能体和统一权限做对。

## 8. 推荐方案

推荐采用：

1. 目标架构：Approach B（领域重构式）
2. 交付策略：Approach C（分阶段兼容式）

换句话说：

1. 不按旧页面逐页照搬来设计数据模型。
2. 但交付时，按旧系统真实使用频率与替换优先级逐步上线。

## 9. 新系统总体架构

## 9.1 前端技术栈

建议：

1. `Vue 3`
2. `TypeScript`
3. `Vite`
4. `Vue Router`
5. `Pinia`
6. `Element Plus`
7. `VueUse`

1. Element Plus 适合作为教师端与后台的基础组件体系。
2. 学生端不应只套用默认 Element Plus 风格，需要单独做主题包装和更具吸引力的视觉层。
3. 主题切换应基于 CSS Variables / Design Tokens，而不是每个页面单独改颜色。

建议：

1. `FastAPI`
2. `SQLAlchemy 2.x`
3. `Pydantic v2`
4. `Alembic`
5. `httpx` 用于外部 LLM 接入
6. `APScheduler` 或后台任务队列用于定时任务/异步处理
7. 文件存储抽象层（本地磁盘优先，后续支持对象存储）

说明：

1. FastAPI 负责 API、鉴权、文件上传、AI 代理、导入导出和系统设置。
2. 所有领域模块都应模块化，不要回到“一个巨型 app.py”。

## 9.3 鉴权与权限模型

不建议保留“教师页”和“管理员页”完全分离的模型。建议统一为：

### 用户身份

1. student
2. teacher

### 权限包

1. `class.manage`
2. `course.manage`
3. `resource.manage`
4. `quiz.manage`
5. `student.manage`
6. `room.manage`
7. `system.manage`
8. `ai.manage`
9. `school.manage`

### 关键设计

1. 管理员不再是独立账号体系，而是拥有 `system.manage` 等权限的教师账号。
2. 支持“教师 + 管理员权限”的合并模型。
3. 为后续多校区、多教研组、多级管理员预留空间。

## 9.4 数据库策略

### 第一阶段

1. 数据库：SQLite
2. 目标：快速起步、方便本地开发与学校内网轻量部署

### 第二阶段

1. 数据库：PostgreSQL
2. 目标：支持更高并发、更可靠事务、更强查询能力

### 设计约束

1. 从第一天开始禁止依赖 SQLite 私有行为。
2. 主键、索引、外键、时间字段、布尔字段都按 PostgreSQL 兼容方式设计。
3. 所有文件路径与业务数据分离。
4. 历史导入链路单独设计，不把旧库结构污染新库。

表现形式：

1. 由教师在学案或课时中创建

3. `scope_id`
4. `provider_id`
5. `model_id`
6. `system_prompt`
7. `welcome_message`
8. `visible_to_students`
9. `visible_to_teachers`
10. `enabled`

1. 密钥不落库明文，必须走环境变量或密钥管理。

4. 学生 AI 会话要有频率限制。

2. 账号、课程体系、班级、学案、任务全部按新模型重新创建。
3. 新旧系统在一段时间内可并行存在，但不做自动数据同步。
4. 首版表结构仍然保留“未来可导入历史数据”的扩展余地，但不为此牺牲当前开发效率。

如果后续某一阶段确实需要导入旧数据，再单独做离线导入程序：

1. 读取旧 SQL Server 数据。
2. 导出为中间 JSON/CSV。
3. 清洗旧字段。
4. 映射到新领域模型。
5. 导入 SQLite 或 PostgreSQL。
